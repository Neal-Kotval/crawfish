# CRA-204 — CL-4: Durable, crash-resumable Refine loops

**What shipped.** `Refine._checkpoint` and a `resume: bool` execute path in
`crawfish/refine.py`, wiring the loop (CL-1) into Crawfish's two durability halves —
deterministic cassette replay (`runtime/replay.py`) and the F-2 composite-key
`ExecutionLedger` (`ledger.py`). A loop that crashes at iteration 3 of 5 now restarts at
iteration 4 re-paying **$0**, instead of re-running from scratch.

**Mechanism.** Each completed iteration's frozen `Output` is checkpointed via
`ExecutionLedger.checkpoint_iteration(loop_id, item_id, edge_id, visit, output_content_sha)`.
`loop_id` is **deterministic** — `compute_loop_id(body.content_sha(), item_lineage,
edge_id)`, never `new_id()` — so two process invocations of the same loop over the same
item re-derive the same id. On `resume=True`, `completed_visits(loop_id, item_id, edge_id)`
is loaded; those visits are re-run through the (replay) runtime, which returns the cached
`RunResult` at **zero cost** (no budget charge on replay). Determinism is therefore
**content-hash-verified**, not trusted: the replayed Output's `produced_by` is the
deterministic `body.content_sha()#visit` coordinate, so its content sha reproduces the
checkpointed reference bit-for-bit.

**$0 resume — how the tests prove it.** `test_crash_then_resume_zero_recompute` records
cassettes for an uninterrupted reference run, simulates a crash (a `max_iters=2` loop that
checkpoints visits {0,1} then dies), then resumes with `resume=True` on the full bound over
the **same ledger**. After resume: `ctx.cost_budget.spent_usd == 0.0` (replayed iterations
charge nothing, and the one fresh iteration's cassette also replays at $0), the resumed
final `Output` sha equals the uninterrupted reference, and `completed_visits == {0,1,2}`.

**Atomicity.** The per-iteration checkpoint is written only after both the body `Output`
and (for a `VerifierStop`) its verdict are in hand — so a crash between the body and
verifier calls leaves no half-checkpointed iteration: resume re-runs that iteration
(replaying $0) rather than skipping the verifier or double-charging.

**Security / tenancy.** Every ledger row carries `org_id`, so a cross-tenant resume cannot
see another org's completed iterations (`test_resume_org_isolation`). Only a frozen `Output`
reference (its content sha) is persisted — never a mutable channel. `raise_if_cancelled`
runs before each iteration, so a cancel between iterations leaves a clean, resumable
checkpoint (`test_cancel_between_iterations_leaves_clean_checkpoint`).

## Public API (extends CRA-202)

- `Refine.execute(seed, ctx, runtime, *, ledger: ExecutionLedger | None = None,
  resume: bool = False, produce=None)` — `ledger` enables durable checkpointing;
  `resume=True` replays committed iterations at $0.
- `Refine._checkpoint(ledger, loop_id, item_id, visit, output)` — persists one iteration's
  frozen Output into the F-2 ledger (atomic over body output + verdict).

**Ledger / hash implications.** No schema change — reuses the F-2 `ledger_loop` namespace
and `compute_loop_id` as-is. New invariant introduced by this issue: a `Refine` iteration's
`Output.produced_by` is the deterministic `body.content_sha()#visit` coordinate (not a
volatile `Run.id`); this is what makes the content-sha resume verification sound. If that
coordinate scheme ever changes it shifts every iteration Output sha (a migration).
