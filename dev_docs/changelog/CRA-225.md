# CRA-225 — AL-DV2: save/recall, a name→hash registry (git for Definitions)

Composition (`with_*`, CRA-224) gives content **hashes** but no **names**. Git's ergonomic is
a *mutable name pointer* into an *append-only, immutable, content-addressed object store*.
Crawfish already had the immutable side (a frozen `Definition` is content-addressed by
`Definition.content_sha`); AL-DV2 adds the name registry and the version log: `save` /
`recall` / `log`, in the new `crawfish.definition_store` module.

## Public API added (`crawfish.definition_store`)

- `DefinitionStore(store: Store, *, org_id="local")` — a Store-backed, append-only,
  org-scoped name→hash registry.
  - `save(name, definition, *, parent=None) -> str` — requires a **frozen** (eval-mode)
    Definition; stores the body content-addressed (dedup), moves the name pointer to the new
    sha (the **sole** mutation), and appends a `DefinitionVersion` lineage event carrying the
    `parent` edge. Returns the content sha.
  - `recall(name, *, sha=None) -> Definition` — resolve the latest (`recall(name)`) or a
    pinned historical version (`recall(name, sha=...)`, or `recall("name@sha")`, or a bare
    content sha). **Pure** — reads a stored object, re-seals it frozen, and **never mints a
    new sha**.
  - `log(name) -> list[DefinitionVersion]` — the append-only lineage, oldest→newest, with
    correct `parent_sha` edges. `head(name) -> str` — the currently-pointed sha.
- `DefinitionVersion` — one append-only lineage point (`name`, `sha`, `version`,
  `parent_sha`, `seq`).
- Errors: `UnfrozenDefinitionError`, `UnknownNameError` (also covers cross-org invisibility),
  `UnreachableShaError` (used by AL-DV3 `reset`).

## Storage planes (all `org_id`-scoped via the `Store` seam)

- **object store** (`kind="definition_object"`, keyed by content sha) — append-only,
  content-addressed; byte-identical Definitions collapse to one stored object (dedup).
- **name pointer** (`kind="definition_name"`, keyed by name) — the one mutable row; `save`
  overwrites it (a branch-tip move).
- **version log** (`kind="definition_version"`) — one append-only event per `save`, keyed by
  a deterministic `(org, name, seq)` id, so two byte-identical saves dedup the object but
  record **two** pointer events.

## Determinism

The pointer move is the sole mutation; the object store is append-only + content-addressed;
`recall` is pure and never mints a sha; tenancy on every row; no model call, no wall clock.
`save("extract", d)` then `recall("extract")` returns the same sha; saving an unfrozen
Definition raises; `recall("extract@<sha>")` resolves a historical pointer after the name
moves on.

## Fork (decision log) — a dedicated `DefinitionVersion`, not `learning.VersionRecord`

The issue says "shares one `VersionRecord` type with `LearningLoop`". `learning.py` is out of
scope for this owner, and its `VersionRecord` carries a mutable `active` flag and eval
`scores` that don't fit a git-style pointer log (where the **name row** is the single mutable
pointer and there are no benchmark scores). A new, purely append-only `DefinitionVersion`
mirrors the *lineage shape* (`sha` + `parent_sha`) without coupling the registry to the
LearningLoop's promotion state. Reconciling them onto one shared lineage type is a follow-up
when `learning.py` is in scope. The discovery-only `Registry` (F-6/OPT-4) is untouched.
