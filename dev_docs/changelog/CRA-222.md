# CRA-222 — OPT-4: dependency resolver + lockfile for summoned units

`craw freeze` (`cli.py`) only hashed already-discovered local units — no resolution. As a
Definition *summons* units by reference (`DefinitionRef.id` at a version constraint), an
unpinned transitive closure breaks replay reproducibility: an un-versioned mutation can
silently enter a frozen run. This adds a pure resolver that walks the closure, picks the
highest compatible version, detects conflicts/cycles, and pins to `(version, content-sha)`.

## Public API added (`crawfish.resolve`)

- `SemVer(major, minor, patch)` — a pure-Python `MAJOR.MINOR.PATCH` comparator (no new
  third-party dep, ADR discipline). `SemVer.parse("1.2")` -> `1.2.0`; a trailing `-sha`
  (as `str(Version)` renders) is captured and ignored for ordering. Ordering is SemVer
  precedence.
- `Candidate(id, version: SemVer, content_sha, dependencies: tuple[DefinitionRef, ...])`
  — one concrete available version of a summonable unit, carrying its own summoned refs so
  the closure is walkable offline.
- `CandidateSource` (`runtime_checkable Protocol`) — the **injected**, offline source of
  candidates: `candidates(unit_id) -> list[Candidate]`. The resolver never reads disk or
  the network itself. `InMemoryCandidateSource` is the default impl (and what tests inject);
  it returns candidates highest-version-first regardless of insertion order.
- `Pin(id, version, integrity)` — one resolved unit; `integrity` is `"sha256:<content_sha>"`.
- `Lockfile(root_id, pins, org_id="local")` — the pinned transitive closure.
  - `closure_sha() -> "sha256:..."` — one hash over the **sorted, de-duplicated** pin set.
    A run embeds this single *reference*, keeping run identity small (vision §5
    reference-by-version). Independent of `org_id` and of pin/insertion ordering.
  - `to_dict()` / `from_dict()` — JSON round-trip; `from_dict` re-verifies the recorded
    `closure_sha` and fails closed on a mismatch (corrupted/hand-edited lockfile).
- `resolve(root: Candidate, source: CandidateSource, *, org_id="local") -> Lockfile` — the
  resolver entry point.
- `read_lockfile(text) -> Lockfile` / `write_lockfile(Lockfile) -> str` — canonical JSON io.
  **Reading a lockfile is data-only; it never executes unit code.**
- `ResolutionError` — raised on every fail-closed condition.
- `LOCKFILE_VERSION = 1` — on-disk schema version; an unknown version fails closed.

## Resolution semantics

- **Ranges (v1):** exact pin, `^` (caret), `~` (tilde). `^1.2` -> highest `1.x >= 1.2`;
  `~1.2` -> highest `1.2.x`; exact `1.2.0` -> only `1.2.0`. For `0.x` the caret treats the
  minor as the breaking axis (`^0.2` -> highest `0.2.x`), matching SemVer convention.
- **Highest compatible:** among candidates satisfying the constraint, `max` by version.
- **Determinism:** each candidate's direct dependencies are walked in sorted `(id, version)`
  order, and the lockfile pins are sorted; identical inputs ⇒ identical `closure_sha` across
  machines.
- **Pure & offline:** no IO, no network, no model call. The `source` is injected.

## Fail-closed conditions (`ResolutionError`)

- Unknown unit (no candidates) — names the requirer + constraint.
- No candidate satisfies a constraint — lists the available versions.
- **Version conflict** — same unit already pinned at a different version by another
  requirer; the message **names both requirers** (acceptance criterion).
- **Dependency cycle** — detected at edge time against the active DFS path (so a back-edge
  into an already-resolved node, e.g. the root, is still caught), names the cycle path.

## Registry reconciliation (per the issue)

The `Registry` (`discovery.py`) stays **discovery-only** (immutable entry points + dir scan,
keyed by `(kind, name)`, no version axis). The resolver does not depend on it; it consumes
the structural `CandidateSource` protocol instead. A future `DefinitionStore` (AL-DV2) or a
registry adapter can implement `CandidateSource` to feed real `(version, content_sha)`
candidates — `discovery.py` was left untouched.

## Tenancy spine

`org_id` (defaulted `"local"`) scopes the recorded closure but does **not** enter the pins
or the `closure_sha`: pins are content-addressed and org-agnostic, so the same closure
resolves byte-identically across tenants (verified by `test_closure_sha_independent_of_org`).

## Determinism / mutation

A mutated summoned unit produces a new `content_sha`, so its pin integrity changes and the
`closure_sha` diverges — an un-versioned mutation cannot enter a previously-frozen closure
without a re-freeze (verified by `test_mutated_unit_gets_new_closure_sha`). A `--check`-style
drift gate is then a `closure_sha` comparison (the CLI command is impl-cli's `craw lock`).

## Acceptance (tests in `test_resolve.py`, 17 cases)

- `^`/`~`/exact range selection (incl. the `0.x` caret rule) picks the highest compatible.
- Transitive closure pins every reachable unit with `sha256:` integrity; a diamond with one
  resolved version contributes a single pin.
- Unknown unit / unsatisfiable constraint / conflict (names both requirers) / cycle all
  raise `ResolutionError`.
- `closure_sha` is independent of candidate insertion order and of `org_id`; a mutated unit
  sha diverges it.
- Lockfile JSON round-trips losslessly; a tampered `closure_sha` and an unsupported
  `lockfile_version` both fail closed.

All tests use pure in-memory candidates — no live model calls, no IO.
