# CRA-207 — C2b: Per-iteration ledger versioning + durable resume

**What shipped.** The durable half of C2, layered onto the C2a `Program` driver
(`workflow.py`): per-iteration program-graph versioning + crash-resumable cycles over the
**F-2 composite-key** `ExecutionLedger`. A `Program` whose cycle crashes mid-iteration now
re-derives the committed iterations at **$0** on `run(..., resume=True)` instead of
re-paying from scratch. This shares the CL-4 (`Refine`) checkpoint substrate — funded once,
durable `Refine`, `Program` loops, `recurse` (C3), and counterfactual replay (R3) all fall
out of the same ledger.

**Mechanism.** Each back-edge traversal: (1) `Output.derive`s a fresh content-addressed
Output and records its `output_content_sha` (F-0) via
`ExecutionLedger.checkpoint_iteration(loop_id, item_id, edge_id, visit, output_ref)`; (2)
the `visit` index is the per-`(item, edge)` cursor. `loop_id` is **deterministic** —
`compute_loop_id(region_version, item_lineage, edge_id)`, never `new_id()` — so a second
process re-derives the same coordinate. On `resume=True`,
`completed_visits(loop_id, item_id, edge_id)` is loaded; those visits replay through the
`RecordReplayRuntime` (cached `RunResult`, **zero budget charge**), and the resumed final
Output sha reproduces the uninterrupted reference bit-for-bit. The ledger stores content-
hash **references**, not embedded Outputs (vision §5 reference-by-version).

**Determinism.** Resume is **content-hash-verified**, not trusted: the replayed Output's
`produced_by` is the deterministic `{region_version}#{edge_id}#{visit}` coordinate, so its
content sha must equal the checkpointed reference. Each iteration appends a distinct
`output_sha`.

**Security / tenancy.** Every `ledger_loop` row carries `org_id`, so a cross-tenant resume
cannot see another org's committed iterations (`test_ledger_rows_carry_org_id_cross_org_
isolation`). `raise_if_cancelled` runs before each iteration, so a cancel between
iterations leaves a clean, resumable checkpoint.

## Public API

- `Program.run(prompt=None, *, ctx=None, runtime=None, resume: bool = False)` — `resume=
  True` loads committed visits from the F-2 ledger and replays them at $0 before any fresh
  work.

**Acceptance test.** `test_program.py::test_durable_resume_recharges_zero` (record + commit
under a shared store, then resume in a second `Program` instance: `inner.calls == 0`,
`spent_usd == 0.0`, resumed final sha bit-identical) and
`test_replay_identical_sequence_and_outputs`.

**Ledger / hash implications.** No schema change — reuses the F-2 `ledger_loop` namespace,
`checkpoint_iteration`/`completed_visits`, and `compute_loop_id` as-is. The new derived
input is `region_version` (folded content shas of the cyclic region's frozen Definitions);
a change to its composition shifts every loop_id (a migration, documented in CRA-206).
