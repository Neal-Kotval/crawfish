# CRA-226 — AL-DV3: modify/reset, git-like checkout over the version log

With names + CoW edits (AL-DV1/2), the remaining git verbs are branch-local mutation
(`modify`) and pointer rewind (`reset`). Both ship as free functions in
`crawfish.definition_store`, composing the `DefinitionStore` verbs — no new persistence law.

## Public API added (`crawfish.definition_store`)

- `modify(store, name, fn) -> str` — `recall → fn → save(parent=old_sha)`. `fn` composes via
  the `with_*` derivation operators (each returns a **new frozen** Definition), so the result
  is already sealed; `modify` saves it with the prior sha as the lineage `parent` edge and
  advances the pointer. Returns the new sha. **Train mode only**: a recalled, frozen
  (eval-mode) name is read-only, so an `fn` that edits in place raises `FrozenError`. Raises
  `UnfrozenDefinitionError` if `fn` returns an unfrozen draft.
- `reset(store, name, to) -> str` — a **pure pointer move** (git checkout). Mints **no**
  object and **no** lineage event; refuses a `to` not in `log(name)` (`UnreachableShaError`);
  reversible. Returns the sha now pointed at.

## Determinism

`reset` mints no content; `modify` routes through the same `with_*`/`_refreeze` content-hash
law as composition (no in-place edit, no model call). Acceptance: `modify` advances the
pointer, records `parent_sha`, leaves the old sha recallable via `@sha`; `reset` rewinds the
pointer, mints nothing, and after it `recall(name)` and the original `@old_sha` return
content-equal Definitions; `reset` rejects an unreachable sha; `modify` on an eval-mode name
(an in-place edit) raises; same start + pure `fn` ⇒ same resulting sha.

## Deferred (as resolved in the issue)

Three-way `merge` is deferred to R1's typed-diff work. `reset` never prunes orphans (it only
moves the pointer), preserving `craw share` reproducibility — orphaned objects stay in the
append-only object store.
