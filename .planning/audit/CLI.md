# CLI — Architecture Audit

Scope: `cli/orgctl/` (org-control MCP server) and `cli/projectctl/` (per-project `.crawfish` engine). Source audited under each `src/`. Compiled output (`dist/`, `dist-test/`) assessed for hygiene only. Cross-tier comparison against `cloud/server` for the board / inbound / budget concepts. Read-only; no source modified.

Anchors read: `.planning/ROADMAP.md` (M3 Phases 12–19), `docs/roadmap/ORCHESTRATOR-STAGES.md` (O0–O7), `docs/roadmap/GRAND_PLAN.md` (§3.16/§3.17), `CLAUDE.md`.

### Strengths

- **MCP dispatch is cleanly extensible.** `index.ts` uses a prefix-dispatch pattern (`name.startsWith("triage_")`, `"planner_"`, `"inbound_github_"`, etc., index.ts:223–248) plus spread `*_TOOL_DEFS` arrays. Adding the M3 craw/orchestrator tools is a matter of one import, one spread into `TOOL_DEFS`, and one `startsWith` branch — no churn to existing tools. The `runTool` wrapper centralizes the `tokens_used`/error envelope (index.ts:253–271), so new tools inherit the contract for free.
- **Per-tool-group modularity.** Each capability (`criteria`, `budget`, `triage`, `planner`, `inbound/*`) exports its own `*_TOOL_DEFS` + `dispatchX`, keeping the registry edit (`index.ts`) small and the per-feature surface owned by one file — aligns with the CLAUDE.md fan-out ownership model.
- **board.ts event model is genuinely event-sourced and forward-compatible.** `BoardEvent` already carries `cycle_id`, `epic_id`, `labels`, `watchers`, `links`, `token_budget`/`token_spent`, and an `activity_log` taxonomy that ORCHESTRATOR-STAGES pre-stage names as the orchestrator's event vocabulary (`task_*`, `budget_breach`, `escalated`). `foldTasks` tolerates malformed lines without failing the whole read (board.ts:113–117). The fold-on-read design is the right substrate for replay (O3.4).
- **Inbound adapters are test-injectable.** `github-issues.ts` injects `runGh`, `budget.ts`/`criteria.ts` inject `fetch` + `lensBase` — clean seams for the integration suites O1.10 / O2 require.
- **projectctl is the more mature CLI:** it ships `lock.ts`, `hash.ts`, `redact.ts`, a verb architecture, and a real test suite — patterns orgctl would benefit from adopting.

### Top 3 risks for roadmap delivery

1. **The "board" is not one thing.** Three uncoordinated task stores (orgctl `board.jsonl`, projectctl `.crawfish/tasks/*.md`, cloud Prisma `Issue`) with no shared schema or sync. ORCHESTRATOR-STAGES O0/O1 assume a single canonical board substrate; Phase 13 intake writes to `board.ts:appendEvent` while the just-built cloud ingestion writes Prisma. Without an explicit canonical-store decision + projection contract, O1 will either fork state or require a mid-stage rewrite. **This is the single biggest threat to M3.**
2. **Write-path incoherence + no locking in orgctl.** board.ts writes fs directly while criteria/budget/preflight proxy through the lens; orgctl has no file lock while concurrent craw dispatch (O3) is on the roadmap. The "single-writer" guarantee in CLAUDE.md is not enforced anywhere in orgctl code. Concurrency bugs here are silent data loss, not crashes.
3. **Inbound duplication hardening into divergence.** GitHub ingestion exists twice with incompatible external-ref keys, and Linear only exists cloud-side. O1.2 explicitly planned to reuse the orgctl inbound pattern; that reuse did not happen, so dedup-across-sources (the same issue via `gh` and via REST) is currently impossible. Every new integration (GRAND_PLAN §3.17 targets 30+) doubles this surface unless one inbound contract is fixed now.

### Concept duplication across tiers (board / inbound / budget): cli vs cloud

**Board — duplicated three ways, unreconciled.**
- `cli/orgctl/src/board.ts`: authoritative append-only `board.jsonl` per org on disk under `~/.crawfish/orgs/<org>/`. Rich event schema, fold-on-read.
- `cli/projectctl/src/project-board.ts` + `.crawfish/tasks/*.md`: a *second*, per-project task representation as markdown frontmatter files. The router and agent-stats operate here — so the routing engine the orchestrator is told to inherit is bound to a different store than the orchestrator's board.
- `cloud/server` Prisma `Issue` table: a *third* store, written by `lib/sync.ts`, scoped to `Project` not `Org`, with its own `provider/externalId` identity. No code path syncs Prisma `Issue` ↔ disk `board.jsonl`.
- Verdict: this is real divergence, not intentional layering. The roadmap (O0/O1) names the disk board as canonical; the cloud `Issue` table should be reframed as a sync projection of inbound issues that lands on the board via `appendEvent`, not as a parallel authority. Needs an ADR before Phase 13.

**Inbound — duplicated and incompatible.**
- orgctl: `gh` CLI transport, `external_ref:{kind:"github_issue",id,url}`, output is a `TaskCreateInput` for the disk board. Also owns the round-trip *mirror* (`mirrorStatusToGithub`) — close/comment/reopen back to the issue.
- cloud: GitHub REST + Clerk OAuth and Linear GraphQL, `provider/externalId/externalKey` identity, upsert into Prisma. No mirror-back.
- The two cannot dedup each other. ORCHESTRATOR-STAGES O1.2 intended cloud to extend the orgctl pattern; instead it is a parallel implementation. Linear inbound exists only cloud-side, so the disk board has no Linear path at all.
- Verdict: consolidate on one normalized inbound shape + one external-ref key. orgctl's `external_ref` + `triage_normalize` is the natural contract; the cloud poller should produce that shape.

**Budget — single logic, but split enforcement surface, and a contract gap.**
- The *decision logic* (`onBudgetBreach`) is cleanly factored as a pure function in orgctl `budget.ts` and is the intended reuse target (ORCHESTRATOR-STAGES O1.6: "wires the existing `cli/orgctl/src/budget.ts` cost-manager pattern"). This is the least-duplicated of the three concepts — good.
- But enforcement is split: the MCP tool is a thin HTTP proxy to the lens, while O1.6 adds a *second* enforcement point in `cloud/server/src/orchestrator/budget.ts` at the worker level. As long as both call the same `onBudgetBreach` pure function, this is acceptable layering rather than duplication.
- Gap: the `"escalated"` status the breach produces is not yet in board.ts's canonical `TaskStatus`, so a breach currently writes a status the board rejects on read/validate. Fix the widening before wiring O1.6.

**Overall:** budget is the model to follow (one pure decision fn, multiple thin enforcement points). Board and inbound are the two concepts that have forked across cli↔cloud and must be reconciled with explicit canonical-store + canonical-inbound-contract decisions before Phase 13 (O1), or M3 inherits three task stores and two GitHub integrations.
