## Cloud — Architecture Audit

Scope: `cloud/server/` (Express + Prisma) and `cloud/platform/` (React + Vite + Clerk SPA). Audited against `.planning/ROADMAP.md` (Phases 1-20, milestones M0-M3) and `docs/roadmap/ORCHESTRATOR-STAGES.md` (O0-O7). Read-only.

The cloud tier is currently a clean, well-factored **Phase 2 + Phase 20** implementation: org/project/issue sync, Clerk-or-dev auth, device-link, invites, GitHub + Linear connectors. It is sound for what it does. The gap is **breadth**: the data model, RBAC, and runtime stack cover ~2 of the 20 phases. M1 (board: cycles/epics/criteria/links/FTS5) and M3 (orchestrator: durable workflows, SSE, audit, billing) each require net-new subsystems, not incremental edits. Several current shortcuts (silent workspace auto-provision, two-value role string, no role gate on writes, sqlite) become liabilities the moment those phases land.

### Findings

| Severity | Area | Finding | Roadmap phase(s) threatened | Recommendation |
|---|---|---|---|---|
| BLOCKER | Realtime | No SSE/WebSocket/streaming anywhere in `cloud/server`. `index.ts` mounts only JSON routers; `express.json({limit:"1mb"})` and CORS are the entire transport story. M3 O3 (`stream.ts`, per-craw SSE) and Phase 15 SC-1 ("real-time per-craw streaming tool-calls over SSE") have zero substrate. Phase 5's "live token-budget bar" also implies push. | 15 (O3), 5, 18 | Stand up an SSE endpoint pattern now (a `res.write`/`text-event-stream` route family + a pub/sub abstraction). Decide single-process vs Redis fan-out before O0. ORCHESTRATOR-STAGES O3.2 assumes "existing lens SSE infrastructure" is reused — but that lives in `desktop/lens`, NOT `cloud/server`; the cloud side must build its own. Flag this assumption as false. |
| BLOCKER | Data model | The schema models **sync mirrors**, not the board domain. There is no `Cycle`, `Epic`, `Task`/`Issue`-with-state-machine, `AcceptanceCriterion`, `TaskLink`, `Activity`/event, `Budget`, or `WorkflowRun`/`Checkpoint`/`AuditLog` model. `Issue` (schema.prisma:152) is a read-only provider mirror (`@@unique([projectId,provider,externalId])`, `syncedAt`), not an authored work item with transitions. M1 Phases 4-7 and all of M3 require these as first-class tables. | 4,5,6,7,12-19 | Treat the board domain model as a dedicated phase-4 design task. Do NOT overload the existing `Issue` mirror to also be the authored task — they have different lifecycles (one is upserted-from-provider, one is human/agent-authored with a status machine). ROADMAP Phase 20 milestone-note already says the cloud `Issue` is "the source of truth that Phase 13's webhook/poller can later feed" — reconcile that intent with a separate authored-task table. |
| BLOCKER | Data model | sqlite is the datasource (schema.prisma:6, comment "swap to postgresql for prod"). FTS5 (Phase 7 SC-2) is a sqlite-specific feature; Postgres uses `tsvector`/`pg_trgm`. The roadmap names "FTS5" explicitly (Phase 7, M1) yet Phase 2 SC-2 names "Postgres + Prisma on Neon" for prod. These are contradictory unless search stays on a sqlite read-replica. Also: `labels String @default("[]")` (schema.prisma:164) is a JSON-in-text workaround for "sqlite has no array/JSON type" — on Postgres this should be `String[]`/`Json`, a migration-shape change. | 7, 2, all M3 | Resolve the engine question as an explicit ADR before Phase 4. If prod is Postgres, "FTS5" in Phase 7 is a misnomer — plan for `tsvector` and rewrite the success criterion. The sqlite→Postgres migration is non-trivial: `labels` text→array, `DateTime` semantics, and the contract harness's `prisma db push --force-reset` (setup.ts:30) all assume sqlite. |
| HIGH | Onboarding | Three independent org-creation paths with divergent semantics, no single source of truth. (1) cloud `OnboardingFlow.tsx:148` → `createOrg` → `POST /api/orgs` (4-question quiz, explicit name, agents). (2) `lib/workspace.ts:69` `ensureUserHasWorkspace` — fires on EVERY authed request (auth.ts:37,120), silently creates an org slugged from the email prefix with `teamSize:"Just me"`. (3) `desktop/dash/web/src/wizards/first-run/index.tsx:14` → dash `lib/orgs` writes an org **to disk**, later mirrored via device-link (`deviceLink.ts:52`). A user who hits the platform first gets a silent auto-org (`alice` workspace); if they then run the cloud quiz they get a SECOND org; the dash first-run makes a THIRD on-disk org that device-link find-or-creates server-side. The "one user, one workspace" comment (workspace.ts:6, Shell.tsx:142) is contradicted by the quiz and by `POST /api/orgs` both still being live. | 1,2,4 | Pick one source of truth. If "one user, one workspace" is the product decision, the cloud quiz should *rename/configure* the auto-provisioned workspace, not create a new org; and `POST /api/orgs` should be gated to legacy/multi-org or removed. The silent auto-provision running on every request (even GETs) is also a write-amplification and surprise-org smell — move it to an explicit first-sign-in hook. |
| HIGH | RBAC | Roles are an unconstrained `String` with **two different vocabularies**: `OrgMember.role` default `"contributor"` // "founder \| contributor \| viewer" (schema.prisma:43) but `Invite.role` is `"owner" \| contributor` (schema.prisma:96, enforced in invites.ts:27). "founder" vs "owner" never reconcile. Almost no route enforces role: `requireMember` (rbac.ts) checks *membership only*, never role; every projects/integrations write is open to any member including a `viewer`. Only `orgs.ts:154` (agent-sync) gates on role, and the dash-sync path. M3 O5 (Phase 17) needs admin/member/viewer with per-resource overrides and a kill switch. | 17 (O5), 4 (Member ACL), 19 | Make `role` an enum, unify the vocabulary (one of founder/admin/member/viewer), and add a `requireRole(min)` helper alongside `requireMember`. Do this before Phase 4 ("Member ACL") — retrofitting role checks across N routes after the board ships is far costlier. The viewer-can-write hole is a real authz bug today, not just future risk. |
| HIGH | Module structure | Orchestrator module (`cloud/server/src/orchestrator/*`, required by O0.2) does not exist, and the current router-registration style won't absorb it cleanly. `index.ts` hand-mounts ~13 `app.use(...)` lines with **load-bearing ordering** (public routes before `authMiddleware` line 56; `linearCallbackRouter` MUST precede auth, comment line 44). Adding orchestrator queue/worker/workflow + webhooks + SSE means more ordering hazards. The dash-sync sub-app is mounted by repeating `dashSyncMiddleware` on 4 lines (50-52), and `/api/orgs/:orgId/integrations` is mounted (line 61) before `/api/orgs` (line 62) — works only because Express matches the more specific path first; fragile. | 12 (O0), 13 | Before O0, refactor to a `registerRoutes(app)` composition (or per-domain route modules with explicit public/authed grouping) so the orchestrator mounts as one cohesive module. The worker/queue belongs OUTSIDE the Express request lifecycle — ORCHESTRATOR-STAGES O0.2 says "extends the existing Express app, not a new service," but a durable worker that survives crashes (Phase 13 SC-4) cannot live in the request process; clarify the deployment topology in ADR-002. |
| HIGH | Auth | Dev shim is keyed only on `NODE_ENV !== "production"` (auth.ts:17), independent of whether a Clerk key is present, and accepts any `X-User-Id` header to mint/login as an arbitrary user (auth.ts:57-63). The doc-comment (auth.ts:48-49) claims the shim runs only when "no CLERK_SECRET_KEY is configured" but the code does NOT check that — the shim always wins in non-prod. Any non-prod deploy (staging) is a full auth bypass via a header. | 2,3,17,19 | Gate the shim on `!IS_PROD && !CLERK_SECRET_KEY` to match its own contract, OR add an explicit `CRAWFISH_DEV_AUTH=1` opt-in. Staging environments that set NODE_ENV=staging would currently be wide open. |
| MED | Data model | No migration exists for the `Integration`/`Issue` models as a reviewable artifact in line with prod. `prisma/migrations/` has `0_init`, `add_project`, `add_integration_issue` — but the contract harness uses `prisma db push --force-reset` (setup.ts:30), which bypasses migrations entirely. On Postgres/Neon, drift between `db push` (dev/test) and `migrate deploy` (prod) is a common foot-gun. | 2, all | Switch the test harness to `prisma migrate reset` so tests exercise the same migration path prod uses. Verify `add_integration_issue` migration matches the current schema.prisma. |
| MED | Security | Integration OAuth tokens are stored plaintext (`Integration.accessToken`, schema.prisma:135-141; comment admits "Tokens are plaintext this phase"). Encryption-at-rest (KMS/libsodium sealed) for `accessToken`/`refreshToken`; never returned in API responses (already respected — integrations.ts:34 omits them). |
| MED | RBAC consistency | The 403→404 collapse is applied inconsistently. `rbac.ts:30` and `orgs.ts:121` collapse non-member to 404 (good, anti-enumeration). Reserve 403 only for authenticated-member-but-insufficient-role, and only on resources whose existence is already known to the caller. |

### Strengths

- **Clean error contract.** `httpError` (errors.ts) gives every route a uniform `{error:{code,message}}` shape; the platform's `formatApiError` consumes it. Consistent, machine-readable, easy to extend.
- **JWT audience separation is correctly designed.** `jwt.ts` uses distinct audiences for `dash-sync` vs `oauth-state` (jwt.ts:14,71) so a device token can never authorize the OAuth callback. Prod fails closed if `JWT_SECRET` is absent/short (jwt.ts:24).
- **Device-link squatter defenses are thoughtful.** `deviceLink.ts:72-78` refuses anonymous mutation of an org that already has a founder; `:code/redeem` (deviceLink.ts:191) blocks founder self-promotion when a founder already exists. The single-use token-then-delete pattern (deviceLink.ts:137-145) is correct.
- **Idempotent provider sync.** `sync.ts` upserts on the compound unique key with token-refresh-on-401 (sync.ts:138-153) and pagination safety caps. The PR-exclusion filter (github.ts:113) and Linear project/cycle-as-label decision (sync.ts:110-117) are pragmatic and documented.
- **`requireMember` is the right shape** for an RBAC primitive — single source of truth, resolves id-or-slug, returns a discriminated result. It just needs a role-aware sibling.
- **Public-vs-authed mounting is deliberate and commented**, and the dash-sync middleware correctly annotates rather than impersonating on user routes (auth.ts:42-46).
- **Contract tests cover the M0 surface** (orgs CRUD+RBAC, invites redeem/mismatch/expiry, device-link, github, linear, issues) matching Phase 3 SC-2 almost exactly.

### Top 3 risks for roadmap delivery

1. **The data model is a sync-mirror, not a domain model — M1 and M3 are net-new schema, not extensions.** Phases 4-7 (cycles, epics, criteria, links, search) and 12-19 (workflows, checkpoints, audit, billing) need ~10 tables that don't exist, plus a status-machine `Task` distinct from the read-only `Issue` mirror. This is the largest single body of unbuilt work and it gates everything downstream. Design it as a dedicated Phase 4 schema task with the sqlite→Postgres decision resolved first.

2. **Realtime is absent and the M3 plan assumes it's already there.** ORCHESTRATOR-STAGES O3.2 says it reuses "existing lens SSE infrastructure" — but that lives in `desktop/lens`, not `cloud/server`, which has no streaming transport at all. Phase 15's headline ("watch craws work in real time over SSE") and Phase 5's live budget bar have no substrate. Building SSE + a fan-out abstraction is a prerequisite spike, not an O3 line item.

3. **RBAC is membership-only with two conflicting role vocabularies and a viewer-can-write hole — and Phase 4 is literally "Member ACL."** The `founder/contributor/viewer` vs `owner/contributor` split, the absence of any role gate on project/integration writes, and the inconsistent 403/404 collapse (which reopens the org-existence oracle on the invites route) all compound. Retrofitting roles after the board and orchestrator ship is far costlier than fixing the enum + adding `requireRole` now, before Phase 4.
